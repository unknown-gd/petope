\section*{Introduction}

When someone asks how can they connect to their server at home,
for example on \href{https://www.reddit.com/r/selfhosted/}{r/selfhosted} subreddit,
I often see that \href{https://tailscale.com/}{Tailscale} is suggested.

Why you may ask? Well to connect to your own server is not as easy task as
to connect normal public facing server.
Most of home networks are behind NAT with periodically changing IPv4 address
\href{https://en.wikipedia.org/wiki/IPv4_address_exhaustion}{since we don't have enough IPv4 addresses},
and there are firewalls that prevent access from outside.

In some cases you don't even have public IPv4 address,
since ISP provided may use \href{https://en.wikipedia.org/wiki/Carrier-grade_NAT}{CGNAT}
which makes connection from the internet impossible even if you use \href{https://en.wikipedia.org/wiki/Dynamic_DNS}{Dynamic DNS}
and open ports on your router.

And most of these problems are solved by Tailscale!
When you start a Tailscale application, it connects to a \href{https://tailscale.com/docs/concepts/control-data-planes}{control plane}
which coordinates your Tailscale network (known as \href{https://tailscale.com/docs/concepts/tailnet}{tailnet}),
tells you which devices are in it, their internal addresses, public key, manages access to each device
and helps you with \href{https://tailscale.com/blog/how-nat-traversal-works}{NAT traversal}
so you can even connect directly to your server.
Instead of classic VPN structure where you connect to relay server to access internal subnet
you connect to each of your servers directly, thus cutting a middle man and an additional latency
and single point of failure\footnote{there still a single point of failure, as tailnet is managed by control plane instead of relay server}.
And since all clients in the network connect to each other directly,
it is called a \href{https://en.wikipedia.org/wiki/Mesh_networking}{Mesh VPN}.

So, with Tailscale you just connect your device to your tailnet, and you can just to your other devices by an address.
No need to add peers to each device, open ports, manage routing. Plug-n-play solution that just works.

Also it makes a great solution when you want to play with your friends together,
but there is much simpler to use alternative named \href{https://www.radmin-vpn.com/}{Radmin VPN},
which has somewhat similiar functionality, but is proprietary, and I just don't like it.

So, after being amazed by how Tailscale works internally, I want to learn by
creating my own VPN solution, learn from my mistakes and while developing everything needed,
and maybe even create something for gamers around the world to use to play together.
And I am totally okay if I won't be able to handle such project, I am sure I will learn a lot of stuff!

\section{Researching existing solutions} \label{existingsolutions}

Since I like \href{https://en.wikipedia.org/wiki/Self-hosting_(network)}{self hosting}
different stuff on my servers, I also like to have control of my data,
particulary don't give a third-party control to it.
That's why I don't fully enjoy Tailscale and Radmin VPN solutions:
\begin{itemize}
  \item Radmin VPN is fully proprietary, so they control the server- and client-side.
        There is nothing stopping them from adding collecting data or even stealing from my PC,
        or adding unauthorized device to my network.
  \item Tailscale has open-source client, but control-plane is fully proprietary and controls
        how client operates. In case of its compomise, your connected devices are fully open
        unless you have configured a firewall. But I trust Tailscale company much more,
        and it feels like a fair trade of security and convinience.
\end{itemize}

But projects like \href{https://syncthing.net/}{Syncthing} and \href{https://yggdrasil-network.github.io/}{Yggdrasil}
inspire me to create something more decentralized, where each device in control of to whom to trust,
and don't rely on single server which commands to whom he can connect and how.

I really liked Yggdrasil approach to the network -
each client is also a server, and an IPv6 address of a client also represents his public key.
This way when you send a packet to an IPv6 address, you can already encrypt message
by using the address, and only client who has right private key can decrypt it.
It is very cool idea since without a single authority you can be sure
that only correct recipient can decrypt and see your messages, and other peers
in the network relay the message until it gets to the right recipient.
Decentralization power, hell yeah!

But without a central server there is no way to find a way how to connect to a peer on the internet.
\href{https://en.wikipedia.org/wiki/BitTorrent}{BitTorrent}
in \href{https://torrentfreak.com/bittorrents-dht-turns-10-years-old-150607/}{old days (before 2005)}
used \href{https://en.wikipedia.org/wiki/BitTorrent_tracker}{trackers}
to find other peers for a torrent you want to download.
The problem is that \emph{distributed file-sharing} relies on \emph{a central server}.
\footnote{which could be improved by using multiple trackers, but it is still not as \emph{distributed}}
To overcome this problem, nowadays \href{https://en.wikipedia.org/wiki/Comparison_of_BitTorrent_clients}{BitTorrent clients}
use \href{https://en.wikipedia.org/wiki/Kademlia}{Kademlia}-based \href{https://en.wikipedia.org/wiki/Distributed_hash_table}{distributed hash table},
aka \href{https://en.wikipedia.org/wiki/Mainline_DHT}{Mainline DHT}.
Basically each peer becomes a distributed tracker, and you stop relying on a single server to find other peers.
This way even if all trackers become offline, you still will be able to find other peers on the internet.
I recommend watching \href{https://www.youtube.com/watch?v=1QdKhNpsj8M}{Kadmelia, Explained} for more in-depth explanation.

\href{https://radicle.xyz/}{Radicle} is another project that aims to bring
p2p to \href{https://en.wikipedia.org/wiki/Git}{Git version control}
which by itself is distributed, but since it is cumbersome to colaborate
in decentralized way, people often use centralized \href{https://en.wikipedia.org/wiki/Forge_(software)}{servers or \emph{forges}}
like \href{https://github.com/}{GitHub} and \href{https://about.gitlab.com/}{GitLab}.
Radicle uses its own protocol named \href{https://app.radicle.xyz/nodes/iris.radicle.xyz/rad%3Az3gqcJUoA1n9HaHKufZs5FCSGazv5}{Heartwood}
which uses a \href{https://radicle.xyz/guides/protocol}{Gossip protocol} to discover other peers and repositories.
For example, imagine this connection between peers:

$$ A \leftrightarrow B \leftrightarrow C \leftrightarrow D $$

So, peer A wants to connect to peer D, but does not know its address.
Peer A \emph{gossips} to peer B, peer B gossips to peer C, and finally peer C returns
response back to peer A. This example shows linear propagation, but in reality a gossip request
propagates through entire network simultaneosly through multiple peers, and each peer stores unique ID
of a gossip request to ignore it next time to prevent a DDoS. This way even without having a direct contact
with peer D, peer A can request information from it.

But also after researching watching the \href{https://www.youtube.com/watch?v=1QdKhNpsj8M}{Kadmelia, Explained} video
I also have started watching other videos from channel \href{https://www.youtube.com/channel/UCdWMU1dBMmB4SrMokGI4jvA}{number 0},
and found out their video \href{https://www.youtube.com/watch?v=RwAt36Xe3UI}{introducing iroh},
a library written in \href{https://rust-lang.org/}{Rust} that does P2P connection between peers effortlessly,
and it just works. After researching this library, it points almost all checks for me, and I would totally use it
for another project that would require such type of networking, or maybe even if it doesn't, just for fun ;)

\subsection{Core functionality} \label{corefunctionality}

After you first launch the app, a crypto key-pair is generated.
Public key is then used as your identifier.
So, to connect both peers together, you just put respective identifier to each peer,
they automatically discover each other and connect. You don't have to put an endpoint to connect,
everything will be handled automatically as it works in iroh.

About IP addresses, if nothing is specified, then they would be cryptographically generated
from identity, otherwise as specified in config.

Also connection failures will be handled on background, if connection to a peer can't be made,
then constant retries will be done. This also applies to network changes, like Wireguard does it,
if an IP of one peer changes, then another peer must send packets to a new destination.

As for protocol, I will be using UDP and then QUIC, since I believe for the start it would

About cryptography, I choose \href{https://en.wikipedia.org/wiki/EdDSA#Ed25519}{ed25519}

\section{Simple UDP server/client}

As my starting point I choose to look into \href{https://doc.rust-lang.org/std/net/struct.UdpSocket.html}{Rust documentation about udp socket},
and wrote following code:

\begin{minted}{rust}
fn server() {
    let socket = UdpSocket::bind("127.0.0.1:3333").unwrap();

    let mut buf = [0; 32];
    let (amt, src) = socket.recv_from(&mut buf).unwrap();

    let buf = &mut buf[..amt];

    println!(
        "received {} from {}",
        str::from_utf8(buf).unwrap_or("raw bytes"),
        src
    );

    buf.reverse();
    socket.send_to(buf, &src).unwrap();
}

fn client() {
    let socket = UdpSocket::bind("127.0.0.1:3334").unwrap();
    socket.connect("127.0.0.1:3333").unwrap();
    socket.send("hello world".as_bytes()).unwrap();

    let mut buf = [0; 32];
    let (amt, src) = socket.recv_from(&mut buf).unwrap();

    let buf = &mut buf[..amt];

    println!(
        "received {} from {}",
        str::from_utf8(buf).unwrap_or("raw bytes"),
        src
    );
}
\end{minted}

\section{Peer discovery}

Now that we can connect via UDP, we need to replace an IP address with an identificator.
For this to work, some sort of a discovery server needs to be in place, which
will tell what IP addresses an identifactor has.

\subsection{Discovery server}

My own implementation will use same UDP protocol,
and messages will consist of simple text strings in format of:
\texttt{command|arguments}.

When a discovery server receives a command, it also receives a source address,
this way after receiving \texttt{register|node id} command discovery server
can save source address like: \mintinline{js}|nodes["node id"] = "source address"|
in a \href{https://en.wikipedia.org/wiki/Hash_table}{hash map}.

\subsection{A node/peer}

A node will just have to register itself on the discovery server, and then retrieve an
address of a node it want to connect to via its node id.
Put this algorithm into a loop, which registers itself, finds target address and sends a message to it.

\subsection{About failures}

At first I have tried connecting both nodes from the same network,
my home Wi-Fi, but both nodes weren't receiving incoming packets.
Probably there is some sort of limitation on the router itself, and
it couldn't decide how to forward packets. So for local network there
must be some other way to find each other.

Luckily there is \href{https://en.wikipedia.org/wiki/Zero-configuration_networking}{Zeroconf}
set of technologies that will allow to broadcast and discover services in local network,
so nodes in the same network can connect to each other even without a discovery server.

\href{https://docs.iroh.computer/connecting/local-discovery}{Iroh}
actually uses mDNS-like system to discover endpoints, not enabled by default
but could be enabled easily.

Also another problem I forsee is that both nodes in \emph{different} networks but under \emph{same} CGNAT
\footnote{aka with same public IPv4 address, but with different internal addresses}
will have also a problem connecting to each other, since discovery server
can't know internal CGNAT address of both nodes.

\section{TCP hole punching}

\href{https://en.wikipedia.org/wiki/TCP_hole_punching}{TCP hole punching} on the paper works
similarly to the UDP hole punching, we just need to send a \texttt{SYN} request from both ends
to the nodes addresses, and next packets should go through the NAT.

The problem is implementing the connection via code, most standard libraries only allow
to TCP socket operate in "connect" or "listen" mode, and "connect" mode only allows to
send packets to \emph{one} destination address.

Luckily, there is a \href{https://crates.io/crates/socket2}{socket2} Rust library
which have OS-dependent API calls to create TCP sockets which could be reused for multiple
destination addresses with same local port, \emph{in theory}. Let's experiment!

\subsection{Trying to reuse same socket}
If you are going to try to create two different \href{https://en.wikipedia.org/wiki/Network_socket}{sockets}
with the same local address, operating system will give a "Address already in use" error.
Thankfully most modern operating systems have an option called \texttt{SO\_REUSEPORT} for a socket
which will allow to have multiple sockets on the same local address. This way we can connect
to a discovery server, register our node id, and then try to connect to another node by reusing
same local address and port.

Simple example of using \href{https://crates.io/crates/socket2}{socket2} library:
\begin{minted}{rust}
use std::io;
use socket2::{Domain, Protocol, Socket, Type};

// Creates a TCP socket with SO_REUSEPORT
fn create_socket() -> io::Result<Socket> {
    let socket = Socket::new(Domain::IPV4, Type::STREAM, Protocol::TCP.into())?;
    socket.set_reuse_port(true)?; // sets SO_REUSEPORT for the socket
    Ok(socket)
}

fn main() {
    // First connect to a discovery server, so OS will give us a local addr
    let first_socket = create_socket().unwrap();
    let addr = "127.0.0.1:4444".parse().into(); // <-- discovery server
    first_socket.connect(&addr.into()).unwrap(); // establish a connection with discovery server

    let local_addr = first_socket.local_addr().unwrap()
    println!("local addr: {}", local_addr); // will print out 127.0.0.1:<random port>

    let second_socket = create_socket().unwrap();
    second_socket.bind(local_addr).unwrap(); // reuse same ip:port as in first socket
    second_socket.listen(128).unwrap(); // run socket in listen mode
}
\end{minted}

Without \mintinline{rust}|Socket::set_reuse_port(true)|, this code would fail with an
"Address already in use" error, which you can try to reproduce by changing \texttt{true} to \texttt{false}.
